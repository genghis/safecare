# SafeCare — Product Plan

## Overview

SafeCare is a secure logistics platform for volunteer food deliveries to at-risk families. It prioritizes recipient physical safety and data privacy through offline-first driver routing, field-level encryption with an off-disk encryption key, blind communications, and aggressive data purge — deployed as a consumer appliance on a Raspberry Pi.

The tradeoff is no hardware-backed key storage, but the key never touches persistent storage.

## Security Architecture

### Off-Disk Encryption Key (DEK)

The Data Encryption Key never touches the filesystem. It exists only in backend process memory (after unlock) and on a physical QR code printout.

| Property | How |
|----------|-----|
| DEK never on disk | Generated by provisioner, shown as QR, never written to .env |
| Seized RPi (off) = safe | Database has ciphertext, no key on filesystem |
| Boot requires human | Admin scans QR to unlock. Unattended reboot stays locked. |
| PII endpoints locked | Return 423 until DEK is loaded via POST /api/setup/unlock |
| DEK canary validation | pgp_sym_encrypt('safecare', DEK) stored in DB, verified on unlock |
| PostgreSQL logging disabled | log_statement=none, log_min_error_statement=PANIC — prevents DEK from leaking into logs |

### Client-Side Encryption (Driver PWA)

| Property | How |
|----------|-----|
| Route data encrypted at rest | AES-GCM-256 in IndexedDB, key derived via HKDF from server session key |
| Key not in persistent storage | sessionStorage only (volatile, tab-scoped, not on disk) |
| Key recovery (online) | GET /api/driver/session-key re-issues from Redis |
| Key recovery (offline) | QR code backup scan |
| Remote wipe | Admin revokes key in Redis, driver poll detects, emergency purge |
| Panic erase | Long-press "Erase" button, instant local destroy, no network needed |
| Shift end purge | flushQueue + purgeAll + clearSessionKey + clearToken + clearTileCache |

### Admin Authentication

| Property | How |
|----------|-----|
| Password | bcrypt (12 rounds) |
| TOTP 2FA | Optional, 6-digit codes, 30-second window |
| Backup codes | 8 single-use recovery codes (bcrypt-hashed in DB) |
| Session tracking | JWT with jti tracked in Redis, 24h TTL |
| Explicit logout | POST /api/auth/admin/logout revokes Redis session |
| Session revocation | revokeAllSessions(adminId) ready for password change endpoint |
| Auth guard | Dashboard redirects to /login if no token, API auto-redirects on 401 |
| Audit logging | login, login_failed, logout, totp_enabled/disabled, system_unlocked, driver_revoked |

### Data Purge

| Data | Retention | Method |
|------|-----------|--------|
| Delivery records (server) | 24h max | Hard DELETE + VACUUM |
| Communication sessions | Hours | Hard DELETE |
| Twilio message logs | Minutes | Twilio Delete API + daily sweep |
| Route data (driver phone) | Shift duration | sessionStorage clear + IndexedDB clear + tile cache clear |
| Audit log | 90 days | Automated daily sweep |

## Remaining Gaps (Prioritized)

### High Priority (security/safety-critical)

| Gap | Description | Effort |
|-----|-------------|--------|
| **Exclusion zones** | Admins draw "avoid" areas (e.g., known surveillance) on the map. OSRM edge-weighting routes around them. Currently zones are delivery-only. | Medium |
| **Communication proxy (blind number pool)** | Drivers and recipients currently see each other's real phone numbers. Need a Twilio proxy so neither side sees the real number. Schema exists, no proxy logic. | Medium |
| **Tailscale networking** | Dashboard accessible to anyone on the local network unless deployed behind the private/public split described in docs/REMOTE-ACCESS.md. | Medium |
| **Route variation** | Same driver gets same route pattern. Spec says vary routes so patterns can't be observed. | Small |

### Medium Priority (usability/hardening)

| Gap | Description | Effort |
|-----|-------------|--------|
| **Remote wipe via push notification** | Current wipe requires driver to poll. Push would be instant even if app is backgrounded. Needs service worker push registration. | Medium |
| **Key rotation tooling** | DEK re-encryption of existing data. `rotate-keys.sh` concept exists but not implemented. | Medium |
| **Docker image pinning** | Some images use `:latest`. Should pin to SHA digest for supply chain protection. | Small |
| **Backup encryption** | pg_dump encrypted with age before storage. | Small |

### Low Priority (polish/nice-to-have)

| Gap | Description | Effort |
|-----|-------------|--------|
| Team gamification | Fun team names for driver morale | Small |
| Route optimization with constraints | Time windows, vehicle capacity, driver preferences | Large |
| Signal export for admin coordination | Manifest as disappearing message | Small |
| Root/jailbreak detection | Hard to implement in PWA | N/A for PWA |
| Hardware-backed key storage (iOS/Android) | Requires native app, not PWA | N/A for PWA |
